## 1. 最高目标

当前最高目标已经收敛为一句话：

**发布一个可解释、可复现、可由玩家在回合末干预的住房市场推演机制。**

这句话拆开后有 5 个硬要求：

1. 固定供给底座下，市场能自然跑起来，不靠人工强推才能成交。
2. 系统能区分“看起来热”和“真的有人抢”，不再把曝光热度误当成竞争热度。
3. 系统能把“该不该进场”和“进场后怎么看、怎么处理、何时行动”拆开处理。
4. 后半段市场即使变薄，也要能解释是自然出清、供需错位，还是玩家补供前的薄市化，而不是黑箱失真。
5. 用户必须能复现测试条件，并在回合末基于清晰信息决定要不要补供、减供、强制挂牌。

## 2. 当前统一口径

### 2.1 对外统一口径：不用“自然月”，统一叫“回合”

从发布口径开始，界面、报告、运行摘要统一使用：

- `回合`
- `虚拟周期`

这里的“回合”不代表现实世界里的自然月。

在本项目中，`1 个回合` 表示一轮完整的市场结算周期，包含但不限于：

- 财务更新
- 激活判定
- 挂牌与调价
- 看房与匹配
- 谈判与下单
- 成交与归档
- 回合末复盘与可选干预

因此：

- `3 回合`、`6 回合` 可以用来比较机制是否稳定
- 但不能直接粗暴等同为现实世界的 `3 个月`、`6 个月`

### 2.2 内部兼容口径：数据库和一部分内部字段仍然保留 `month/months`

当前已经确认：

1. **对外显示层已经大范围改成“回合/虚拟周期”**
2. **内部数据库、配置键、事件载荷、旧脚本参数仍然大量保留 `month/months`**

这不是遗漏说明，而是当前发布阶段的明确边界：

- 当前优先保证发布机制成立、证据成立、用户入口清楚
- 不在发布前做全库字段迁移

## 3. 4 月 15 日以来已实装并生效的机制

### 3.1 固定供给底座与可复现输入链

当前发布链已经不再允许用户随意造一份不可复现的供应盘，而是统一从固定快照出发。

已生效能力：

1. 固定供应盘已锁定为梭子型样本：
   - `spindle_minimum = 39 套`
   - `spindle_medium = 91 套`
   - `spindle_large = 164 套`
2. 启动入口统一改成：
   - 先选固定供应盘
   - 再设需求侧倍率
   - 再决定是否加入自动冲击
3. 需求倍率允许在 `0.10x - 2.00x` 间设置
4. 系统会自动保障两件事：
   - 每类买家画像都不会被压到消失
   - 每类供应画像都仍有对应买家覆盖

机制目的：

- 让测试真正可复现
- 让用户调的是“需求压力”和“冲击”，不是同时随机改掉供给结构

### 3.2 checkpoint 与断点续跑

已生效能力：

1. 每个回合结束后自动落 checkpoint
2. 支持从已有数据库继续往后跑
3. 能区分“历史已完成部分”和“续跑新增部分”

机制目的：

- 降低长测成本
- 让长测、续跑、对照试验能基于同一条主链复核

### 3.3 新买入房产 12 个周期内禁止再次出售

已生效能力：

1. 房子会记录买入时点
2. 刚买入的房子在锁定期内不能再次挂牌
3. 恢复数据库后这个状态仍然保留

机制目的：

- 阻止短期倒手制造伪活跃
- 防止系统靠异常快的周转制造假繁荣

### 3.4 曝光热度与竞争热度彻底拆开

这是本轮最核心的修复之一。

已经确认并生效的链路：

1. 热度已做四层拆分，不再把“被很多人看到”直接当成“很多人真的在抢”
2. 假热房加入硬门槛：
   - 假热房不能再触发“热盘提价”
   - 假热房不能再触发“自动竞价”
3. 真竞争才允许触发更积极的竞价和强制拍卖
4. 反事实信用分配已经接通，不再把本该属于替代房的热度全部记到头部房源
5. shortlist 多样性约束已经接通，避免同一桶里始终只有头部 1-2 套房被看见

机制原理：

- “很多人点开看过”不等于“很多人真的愿意为它加价”
- 只有在真实竞争出现时，才应该允许价格上冲和拍卖升级

机制目的：

- 降低假热
- 保留真竞争
- 让价格上涨更像真实市场，而不是推荐算法误判

### 3.5 真热簇扩散：兄弟房承接与竞败需求外溢

在上一步区分出真竞争之后，又补上了热度扩散链。

已生效机制：

1. 同层兄弟房承接：
   - 真热房周边、同圈层、同替代关系的房源，可以获得有限承接热度
2. 竞败需求外溢：
   - 买家在真实竞价中输掉后，不是凭空消失，而会短时保留更高优先级，转向相近替代房

机制目的：

- 把“真热只集中在 1 套房”扩成“同圈层几套房一起被承接”
- 避免市场永远只有单点热，而没有替代链路

### 3.6 生命周期标签 + 结构化时点角色 + 代码维护连续性

这是“1 号问题”的主链，目前已经落地。

当前结构是三层：

1. **第一层：代码只维护生命周期标签**
   - 只负责给出本回合背景，不直接决定买卖
2. **第二层：LLM 输出本回合结构化时点角色**
   - 立即买
   - 立即卖
   - 先卖后买
   - 有需求但本回合不进场
   - 继续观望
3. **第三层：代码只维护链条连续性与规则约束**
   - 续航
   - 冷却
   - 锁定
   - 资金与交割约束

机制目的：

- 先回答“该不该进场”
- 再回答“进场后看见什么、怎么处理、何时行动”
- 避免代码替 LLM 决定买卖角色

### 3.7 `smart / normal` 不再走人格研究路线，只做行为修饰器

当前确认口径：

- 生命周期是主驱动
- `smart / normal` 只是同一决策窗口内的行为修饰器

已接通的四层差异：

1. 信息可见性
2. 信息处理方式
3. 行动时点
4. 行为稳定性

机制目的：

- 保留现实中的“信息差、反应差、执行差”
- 不把线程带去泛化人格研究

### 3.8 后半段供给续航：卖方再唤醒、旧房释放、回合末玩家干预

当前有效方案已经从“系统自动补尾盘”改成了更符合项目目标的三件事：

1. **价格推高后的卖方再唤醒**
   - 但必须带冷却回合，不允许刚成交完马上被重新点燃
2. **置换链旧房释放**
   - 释放的是旧房，不放开新买房的锁定期
3. **回合末暂停面板**
   - 当交易明显下滑、缺口桶扩大、`NO_ACTIVE_LISTINGS` 偏高时，回合结束后暂停
   - 给玩家看热销画像、偏冷画像、成交套数、缺口桶和找不到在售房的情况
   - 由玩家决定：
     - 定向增供
     - 自动补供
     - 全量补充
     - 减供
     - 强制挂牌

明确不采用的方案：

- **尾盘陈旧审查与再利用**：当前已放弃，不作为主方案

机制目的：

- 不是为了强行把后半段炒热
- 而是为了避免市场过早变成“还有人想买，但找不到合适在卖的房”

## 5. 当前最重要的结论

### 5.1 现在已经不是“系统跑不起来”

当前最关键的结论是：

**不加外部调节，系统也能完整跑完 6 回合。**

这意味着：

- 当前主链已经不是“离开人工强推就失灵”
- 发布时可以理直气壮地说：系统具备自然运行能力

### 5.2 供给干预的价值，不是简单堆高总成交

当前对比已经很明确：

- 含预设干预样本总成交：`73`
- 纯自然 clean baseline 总成交：`75`

两者总成交差距并不大。

真正的差别在后半段：

- 纯自然样本更容易进入“还有人想买，但买不到对口在售房”的薄市状态
- 含干预样本能更明显缓和这种后半段供需错位

因此当前发布口径应该明确说：

**补供/强制挂牌/玩家回合末干预的价值，不主要是为了把总成交拉得更高，而是为了缓和后半段供需错位，让市场不要过早变成薄市。**

### 5.3 后半段成交变少，不等于模型错了

固定供给市场天然会出现：

- 房子逐步卖掉
- 主流流动性好的房源先被吃掉
- 剩余房源和剩余需求的匹配度下降

因此，后半段成交下降本身是合理的。

真正要避免的是：

- 不是自然出清
- 而是过早进入“局部还在提价，但大多数人已经买不到对口在售房”的失真薄市

### 5.4 当前价格形状已经更接近真实市场

当前已经确认：

- 后半段确实还有局部提价
- 但不是全市场一起变成更强拍卖

更准确的说法是：

**后半段提价有，但只是局部增强，不是全市场增强；成交少也不是纯需求塌了，而是有效供给错位后，竞价面收缩了。**

### 7.4.1 为什么金字塔型中样本的后半段供给续航明显强于梭子型中样本

这次对比里，最重要的发现不是“金字塔型成交更多”，而是：

1. **两者总成交占比几乎一样**
   - `G1`：`40 / 91 = 44.0%`
   - `G2`：`41 / 93 = 44.1%`
2. **但后半段“想买却没房可买”的强度差别非常大**
   - `G1`：总计 `156` 次
   - `G2`：总计 `43` 次
3. **而两者总房源只差 2 套**
   - `G1`：`91`
   - `G2`：`93`

这说明真正起作用的不是“多了几套房”，而是“房放在了哪里、后半段剩下了什么”。

#### 先把“是什么”摊开说清楚

如果不把“梭子型是什么、金字塔型是什么、买家画像是什么”摊开，这一节就会只剩抽象判断。这里先把最关键的“是什么”写清楚。

##### 1. 梭子型和金字塔型，配置上到底差在哪里

这部分不在 `.db` 里，首先来自固定供应盘目录：

- `D:\GitProj\visual_real_estate\config\supply_snapshot_catalog_v1.yaml`

当前两类结构的核心定义是：

1. `spindle`（梭子型）
   - 设计目的：让中段改善和主流需求更厚，两端保留但不过度堆积
   - 价格带配比：
     - A 区：低 / 中 / 高 = `0.25 / 0.50 / 0.25`
     - B 区：低 / 中 / 高 = `0.45 / 0.40 / 0.15`
   - 桶位缩放：
     - starter = `0.95`
     - improve = `1.15`
     - luxury = `0.65`
2. `pyramid`（金字塔型）
   - 设计目的：让刚需和入门盘更厚，高端盘更薄
   - 价格带配比：
     - A 区：低 / 中 / 高 = `0.40 / 0.40 / 0.20`
     - B 区：低 / 中 / 高 = `0.55 / 0.35 / 0.10`
   - 桶位缩放：
     - starter = `1.20`
     - improve = `0.95`
     - luxury = `0.45`

一句话解释：

- 梭子型：把供给厚度更多放在中段改善层
- 金字塔型：把供给厚度更多放在入门层和外围承接层

2. **真正决定后半段是否迅速薄市化的，是主流可交易桶位有没有被提前吃空**
3. **如果主流可售层太薄，市场就会出现“局部热点仍在、整体却越来越难成交”的形状**

这对现实住房市场推演的启发是：

1. 不能只看“总房源够不够”
2. 要看“刚需 / 主流改善 / 外围承接层”这些桶位是不是还有可交易库存
3. 如果一个城市的新增或存量供应过度集中在中段改善盘，而主流入门盘和外围承接盘偏薄，就更容易在后半段出现：
   - 局部价格仍坚挺
   - 买方需求没有消失
   - 但交易面迅速收窄

3. 发布时对外要清楚说明：

   - 金字塔型更适合看“刚需和承接层更厚”时的后半段流动性表现

#### 边界说明

这条结论有现实指导意义，但仍然只是**结构方向上的指导**，不是现实自然月预测：

1. 本项目里的“回合”是虚拟市场周期，不等于现实自然月
2. 当前 `G1/G2` 仍是固定供应长测，不包含真实城市里持续新增供给的全部复杂性
3. 因此，这组结果最适合用于回答：
   - 供给结构会不会改变后半段的市场形状
   - 玩家为什么需要在不同供应盘之间切换对照
4. 不适合直接外推成：
   - “现实 6 个月后一定会怎样”
