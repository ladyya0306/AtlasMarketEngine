# 中国住房市场推演发布说明（2026-04-18）

这份说明的目标只有一个：

**用尽量少的术语，把现在这个系统到底能做什么、为什么值得发布、怎么复现，说清楚。**

---

## 1. 这个系统现在是什么

这是一个可以自己跑起来、也允许玩家在关键节点干预的住房市场推演系统。

它不是简单随机出几条交易记录，而是把一轮市场里的几个关键动作串起来：

1. 居民财务变化
2. 谁在这一轮真的会进场
3. 房子怎么挂牌、怎么调价
4. 买家看什么、选什么、怎么谈
5. 哪些交易真的成交，哪些失败
6. 一轮结束后，玩家要不要补供、减供、强制挂牌

这里统一叫“回合”，不叫“自然月”。

原因很简单：系统里的一个回合，不只发生一件事，而是包含一整轮市场结算、撮合、谈价、成交和归档。  
所以它更接近“完整市场周期”，而不是现实日历上的一个自然月。

---

## 2. 现在已经解决了什么

### 2.1 不再把“看起来热”误当成“真的在抢”

以前系统最容易出错的地方，是把“很多人看过”当成“很多人真的愿意加价去抢”。

现在这条已经拆开了：

- 只是曝光高，不会自动推高价格
- 只有真竞争出现时，才允许更强竞价和拍卖升级

这一步解决的是“假热”问题。

### 2.2 不再把“长期需求”和“本回合真要买”混成一团

现在系统已经把三件事拆开：

1. 这个人这回合该不该进场
2. 如果进场，他是立即买、立即卖、先卖后买，还是继续观望
3. 进场之后，他看见什么、怎么处理、什么时候行动

这样做的好处是：

- 不会再把“长期有住房需求的人”都算成当下买压
- 也不会靠代码替人决定“你这个月就该卖房”

### 2.3 玩家已经有真实的回合末干预入口

现在不是只有系统自己跑。

如果某一回合结束后，系统判断市场明显变薄，比如：

- 有明显热销画像
- 缺口画像在扩大
- 很多人想买却找不到正在卖的对口房

系统会暂停，让玩家看到清晰复盘，再决定：

- 定向增供
- 自动补供
- 全量补充
- 减供
- 强制挂牌

这一步已经不只是界面按钮，而是会真实写入结果数据库，并影响下一回合。

## 3. 现在最重要的测试结论

### 3.1 市场在不加外部调节时，也能自己完整跑完

这说明系统已经不是“一离开人工推一把就不行”。

当前最关键的纯自然样本：

- `D:\GitProj\visual_real_estate\results\release_longtest\release_longtest_cleanbaseline_20260418_084908_spindle_large_m6_d1_s606\smoke_report.md`

这条样本显示：

- 一共跑完 6 回合
- 全部 checkpoint 都存在
- 总成交 75 套
- 而且中途没有预设冲击

### 3.2 补供的真正价值，不是单纯把总成交堆高

当前已经做过一组很关键的对比：

- A 组：纯自然运行
- B 组：中途带轻量供给调节

这两组总成交差别不大：

- 纯自然：75
- 带调节：73

但后半段差别很明显：

- 纯自然更容易出现“还有人想买，但没有对口房在卖”
- 带调节时，这种薄市化会明显缓和

所以，补供和强制挂牌的真正意义不是“把数字做得更高”，而是让后半段不要太早变成薄市。

### 3.3 这套系统已经具备可复现入口

现在用户不是随便手填一堆参数，而是按固定入口操作：

1. 先选固定供应盘
2. 再设需求倍率
3. 再决定是否加冲击
4. 再跑回合
5. 在回合末决定要不要干预

而且系统会自动保障：

- 每类买家画像不消失
- 每类房源都仍有对应买家覆盖

### 3.4 关键对照测试已经跑齐

截至 2026-04-18，发布前最关键的对照测试已经补齐：

1. 样本量桥接
   - `spindle_medium`
2. 供给结构敏感性
   - `pyramid_medium`
3. 需求压力方向
   - `buyer_market`
   - `seller_market`
4. 轻量冲击韧性
   - `balanced_resilience`

这些测试现在回答的是四个问题：

1. 缩到中样本后，这套机制会不会直接跑坏
2. 供给结构从梭子型换成金字塔型后，后半段市场形状会怎么变
3. 把需求往冷、热两个方向拉时，市场会不会按常识偏向买方或卖方
4. 加入轻量冲击后，系统还能不能给出可解释结果

当前结论是：**都能跑通，而且结果方向基本符合预期。**

### 3.5 `seller_market` 的卖方强势，不等于“所有房子一起暴涨”

这是当前最容易被误解的一点，所以必须单独说。

在 `seller_market` 对照里，市场确实更偏卖方，但它不是用“所有房子的平均总成交价都更高”来体现的。

更真实的表现是：

1. 多人争抢的房子更多了
2. 热点房更接近挂牌价，部分房子甚至高于挂牌价成交
3. 但同时，整条样本卖掉了更多低总价主流盘
4. 所以“平均总成交价”会被成交结构往下拉

当前最该引用的证据不是单看一个均价，而是一起看下面几项：

1. `seller_market` 的 `Outbid` 失败总数：`89`
   - 自然基准只有 `41`
2. 多人争抢并最终成交的房源：
   - `seller_market`：`18` 套
   - 自然基准：`13` 套
3. 这些多人争抢成交房源的成交价 / 挂牌价均值：
   - `seller_market`：`1.0209`
   - 自然基准：`0.9957`
4. 但整条 `seller_market` 样本里，入门盘成交占比从自然基准的 `14.7%` 升到了 `32.3%`

这说明：

1. 竞价和拍卖式分流是真的发生了
2. 卖方市场机制是生效的
3. 只是卖方强势主要集中在局部热点房源上
4. 而整体成交结构，又明显下沉到了更多低总价主流盘

**现在这个系统已经能在固定供给底座下自然跑起来，能分清假热和真竞争，能把长期需求和本回合有效需求拆开处理，也允许玩家在回合末根据清晰复盘决定是否补供。**

## 5. 当前边界也要讲清楚

### 5.1 “回合”不等于现实自然月

这是当前发布时一定要说清楚的专业边界。

本项目里：

- 回合 = 一轮完整市场周期
- 不等于现实日历月份

所以 3 回合、6 回合更适合用来比较机制是否稳定，不能粗暴直接翻译成现实世界的 3 个月、6 个月。

### 5.2 内部数据库还保留 `month/months`

对外显示已经大范围改成“回合”，但内部数据库和部分兼容字段还保留 `month/months`。

这是当前有意保留的兼容边界，不是漏改。

发布前不做全库字段迁移，是为了避免高风险回归，先把机制和证据稳定下来。
